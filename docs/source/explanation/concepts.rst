Core Concepts
=============

This page explains the fundamental ideas behind CIM and how they fit together.

Workspace
---------

A **workspace** is a self-contained directory that holds everything needed for
a project: cloned git repositories, toolchains, downloaded files, Python
virtual environment, generated Makefile, and configuration files.

A workspace is identified by a ``.workspace`` marker file at its root. It
typically contains:

.. code-block:: text

   ~/dsdk-my-target/
   ├── .workspace          # marker file
   ├── sdk.yml             # manifest (copied from target)
   ├── os-dependencies.yml # OS deps (if defined)
   ├── python-dependencies.yml
   ├── Makefile            # generated by cim makefile
   ├── .venv/              # Python virtual environment
   ├── toolchains/         # downloaded toolchains
   ├── repo-a/             # cloned git repos
   ├── repo-b/
   └── .sdk/               # sentinel files for install targets

Workspaces are isolated — everything lives in the workspace folder except host
OS packages and the shared mirror. Deleting the workspace directory removes
everything cleanly.

Mirror
------

The **mirror** is a local cache (default: ``$HOME/tmp/mirror``) that stores:

- Git bare repository mirrors (used for faster subsequent clones)
- Downloaded toolchain archives
- Cached ``copy_files`` artifacts

The mirror is shared across workspaces. When you initialize a second workspace
that uses the same repository, CIM clones from the local mirror instead of
fetching from the network again. This saves bandwidth and enables offline
operation.

Disable mirroring per-command with ``--no-mirror`` or globally in the
configuration file.

Target
------

A **target** is a predefined SDK configuration inside a manifest repository.
Each target is a directory under ``targets/`` containing at minimum an
``sdk.yml`` file.

.. code-block:: text

   my-manifests/
   └── targets/
       ├── optee-qemu-v8/
       │   ├── sdk.yml
       │   ├── os-dependencies.yml
       │   └── python-dependencies.yml
       └── adi-linux/
           └── sdk.yml

Targets are what users select with ``cim init --target <name>``.

Manifest Repository
-------------------

A **manifest repository** is a git repository (or local directory) that
contains one or more targets.

Organizations typically maintain:

- An **internal** manifest repository with all SDKs and projects
- An **external** manifest repository for customers and partners

CIM discovers targets by scanning the ``targets/`` directory of the manifest
repository. Point CIM at a manifest repository with ``--source``.

Manifest
--------

The **manifest** is the set of YAML files that describe a target. The primary
file is ``sdk.yml``, which defines:

- Git repositories to clone
- Build, test, clean, and flash commands
- Toolchains to download
- Files to copy
- Variables for overridable settings
- Custom install targets

Two optional companion files define host dependencies:

- ``os-dependencies.yml`` — system packages per OS and distribution
- ``python-dependencies.yml`` — Python packages organized into profiles

See :doc:`/reference/manifest` for the complete field reference.

The workflow in detail:

1. ``cim list-targets --source <repo>`` discovers available targets.
2. ``cim init --target my-sdk`` clones repos (via mirror), copies the manifest
   files, and sets up the workspace.
3. ``cim install`` downloads toolchains, OS packages, and Python dependencies.
4. ``cim makefile`` generates a Makefile from the manifest.
5. ``make sdk-build`` / ``make sdk-test`` runs the build.
